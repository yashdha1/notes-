The Compose file, or `compose.yaml` file, follows the rules provided by the [Compose Specification](https://docs.docker.com/reference/compose-file/) in how to define multi-container applications. This is the Docker Compose implementation of the formal [Compose Specification](https://github.com/compose-spec/compose-spec). 

A single docker compose file starts the containers, volumes, networks everything. 
The `docker compose up` command starts the `frontend` and `backend` services, creates the necessary networks and volumes, and injects the configuration and secret into the frontend service.

The primary purpose of Docker Compose is to streamline the entire application lifecycle—building, starting, stopping, and viewing the status of all components with simple CLI commands like `docker compose up` and `docker compose down`.

-> Usual structure in the compose file structure : defining a collection of interoperable _services_, _networks_, and _volumes_ . ie **COMPONENTS** of the yaml files. 

1. **Services** : Services are the core of the Compose file. Each service defines a specific component of the application (e.g., a web server, a database, a cache) and specifies how its container should run. Defines a structure for each and every container. 
	does all of this shit: 
	- **`image`**: Specifies the Docker image to use for the service (e.g., `postgres:latest`).
	- **`build`**: Instead of using a pre-built image, this defines the configuration to build an image from source using a [Dockerfile](https://docs.docker.com/reference/dockerfile/).
	- **`ports`**: Maps ports from the container to the host machine, making the service accessible externally.
	- **`volumes`**: Mounts host paths or named volumes into the container for persistent data or code synchronization.
	- **`environment`**: Sets environment variables within the container.
	- **`depends_on`**: Defines dependencies between services, ensuring a database starts before the application that connects to it.

2. **Networking** : Does the networking shit itself. You need not do this manually. Compose automatically creates a default network for all services to communicate with each other in an isolated environment. You can also define custom networks with specific configurations. 
3. **Volumes**: Named volumes are defined at the top level to manage persistent data that outlives the containers themselves. also can do bindmounts that are managed by the user instead of the docker itself. 
4. - **`.env` files**: Configuration can be made portable across different environments or users by utilizing environment variables, often stored in a `.env` file that Compose automatically loads.

Troubleshooting and logging : 
1. *docker-compose --verbose up* => This command starts all services with verbose logging, showing each step of the process, including network creation, volume mounting, and container startup. It helps pinpoint where the failure occurs.
2. *docker compose logs <service_name>* => just check thy logs.

# Docker Compose & Dockerfile — Quick Reference Notes

## Core Distinction

||Dockerfile|Docker Compose|
|---|---|---|
|Purpose|**Build** an image|**Run** containers|
|Multi-service|❌|✅|
|Layer caching|✅|✅ (delegates to Docker)|
|Multi-stage builds|✅|✅ (via Dockerfile)|
|BuildKit support|✅|✅ (automatic, modern Docker)|

> **Rule of thumb:** Dockerfile = _what_ to build. Compose = _how_ to run it together.

---

## Setup: Node + React + Redis + Postgres

You only write **Dockerfiles** for services you build:

- `backend/Dockerfile` — Node.js app
- `frontend/Dockerfile` — React app

For Redis and Postgres, use **official images** directly in Compose.

---

## Optimised Multi-Stage Dockerfile (Backend Example)

```dockerfile
# syntax=docker/dockerfile:1.6

# Stage 1: Install prod deps only
FROM node:20-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

# Stage 2: Build
FROM node:20-alpine AS build
WORKDIR /app
COPY . .
RUN npm run build

# Stage 3: Minimal runtime image
FROM node:20-alpine
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY --from=build /app/dist ./dist
CMD ["node", "dist/index.js"]
```

**Why this works:**

- Deps layer is cached unless `package.json` changes
- Build artifacts are copied, not the full build environment
- Final image is tiny — no dev tools, no source

## docker-compose.yml (Production)

```yaml
version: "3.9"

services:
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    image: myapp-backend:latest
    environment:
      - DATABASE_URL=postgres://user:pass@db:5432/app
      - REDIS_HOST=redis
    depends_on:
      - db
      - redis
    restart: unless-stopped

  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    image: myapp-frontend:latest
    ports:
      - "80:80"
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    volumes:
      - redis-data:/data

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: app
    volumes:
      - pg-data:/var/lib/postgresql/data

volumes:
  redis-data:
  pg-data:
```

## Layer Caching in Compose

Compose calls `docker build` under the hood — caching behaviour is **identical** to running `docker build` manually.

- `package.json` unchanged → deps layer served from cache
- Source code changed → only layers after the `COPY . .` rebuild

No special config needed.

---

## BuildKit

Modern Docker Compose uses BuildKit **automatically**.

To verify:

```bash
docker buildx version
```

To force-enable (older setups):

```bash
export DOCKER_BUILDKIT=1
```

### Advanced BuildKit in Dockerfiles

```dockerfile
# Cache npm packages across builds
RUN --mount=type=cache,target=/root/.npm \
    npm ci
```

### Remote cache in Compose

```yaml
build:
  context: .
  cache_from:
    - type=registry,ref=myapp-backend:cache
  cache_to:
    - type=registry,ref=myapp-backend:cache,mode=max
```

---

## What Compose Cannot Do

- ❌ Optimize image size
- ❌ Define multi-stage build logic
- ❌ Replace Dockerfile

Compose is an **orchestration layer**, not a build tool.

---

## When to Use What

|Need|Tool|
|---|---|
|Custom OS / install packages|Dockerfile|
|Multi-stage build / image optimization|Dockerfile|
|Run app + DB + Redis together|Docker Compose|
|Env-specific config (dev/prod)|Docker Compose|
|Scale services|Docker Compose|
|Both|Use both (most common)|

---

## For Larger Production

Docker Compose suits: local dev, staging, single-host prod.

For bigger scale → **Kubernetes** or **Docker Swarm**.
