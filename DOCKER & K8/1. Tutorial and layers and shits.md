*Registry vs Repository*

While you're working with registries, you might hear the terms _registry_ and _repository_ as if they're interchangeable. Even though they're related, they're not quite the same thing.

A _registry_ is a centralized location that stores and manages container images, whereas a _repository_ is a collection of related container images within a registry. Think of it as a folder where you organize your images based on projects. Each repository contains one or more container images.

 ### 1. Image are immutable snapshot of the containers :
these layers are stacked on top of one another, and each layer is immutable (once created, it cannot be changed). When Docker creates an image, it doesn't rebuild the entire image from scratch each time. Instead, it reuses layers that have already been created, which speeds up the build process and reduces disk space usage.
Docker internally uses layer caching mechanism to avoid rebuilding layers unnessacarily If a layer has already been built (and its contents haven’t changed), Docker can reuse that layer instead of creating it again.

### 3. Multistage Builds 
When you build an app (say Go or Java), you need a bunch of build tools — compilers, SDKs, package managers. But to **run** the app you need none of that. Just the final binary/output. 
proper multistage building strategy can reduce the sixe of builds from several gbs to single digit mb file. 500X improvement. 
Using named Build stages for the later build to be refercenced later in the docker file for the lighter or lesser image in diffrence enviorment. So AS syntax make the stages reusable.  ex
``` 
FROM node:18 AS builder
.
...later reference it down the line...
in multistage builds....
.
COPY --from = <Stage name>
```
```
# Base build stage
FROM node:18 AS base
WORKDIR /app
COPY package*.json .
RUN npm install

# Build stage
FROM base AS build
COPY . .
RUN npm run build

# Production stage
FROM node:18-alpine AS production
WORKDIR /app
COPY --from=build /app/dist ./dist
CMD ["node", "dist/index.js"]
```

here the simplest flow is : 
1. `base` installs dependencies once.
2. `build` reuses `base`.
3. `production` copies only the compiled output from `build`.
4.  Final image is smaller because it doesn’t include build tools.

### 4. Caching and docker features : 

every instruction in docker creates a layer. and each layer is cached, Reusable and Shared across containers. 
Optimization techniques : 
Each layer has a **cache key** based on: 
	1. The Dockerfile command (`RUN`, `COPY`, `ADD`, etc.)
	2. The command's arguments
	3. The parent layer's cache key
	4. **For `COPY`/`ADD`**: The checksum of copied files

A cache miss in the dockerFIles lead to the rerun and rebuild of the entire docker images for this.

### For better optimisations and stuff docker even has a garbage collector build into it. 
1. GC policies define a set of rules that determine how the build cache is managed and cleaned u.
2. These policies include criteria for when to remove cache entries, such as the age of the cache, the amount of space being used, and the type of cache records to prune.
3. Each GC policy is evaluated in sequence, starting with the most specific criteria, and proceeds to broader rules if previous policies do not free up enough cache. This lets BuildKit prioritize cache entries, preserving the most valuable cache while ensuring the system maintains performance and availability.

* The default GC policies are (approximately):
	1. Remove cache that can be easily regenerated, such as build contexts from local directories or remote Git repositories, and cache mounts, if hasn't been used for more than 48 hours.
	2. Remove cache that hasn't been used in a build for more than 60 days.
	3. Remove unshared cache that exceeds the build cache size limit. Unshared cache records refers to layer blobs that are not used by other resources (typically, as image layers).
	4. Remove any build cache that exceeds the build cache size limit.

## 5. Docker volumes Persistent Storage and Bind Mount  

-  A Docker **volume** is a managed storage area created and controlled by Docker. It exists outside the container’s filesystem but can be attached to one or more containers.
- **Location:** Stored in Docker’s storage area (usually `/var/lib/docker/volumes/` on Linux).
- **Use Cases:**
    - Persist container data across container restarts or deletions.
    - Share data between multiple containers.
    - Store data for databases or apps that need to survive container removal.
        
- **How to create and use:**
    docker volume create mydata  
    docker run -d -v mydata:/app/data myimage
	    Here, `mydata` is a Docker volume, mounted inside the container at `/app/data`.
    

**Advantages:**
- Managed by Docker (you don’t worry about filesystem paths).
- Works across platforms.
- Supports features like volume drivers for cloud storage.


 **2. Bind Mounts**

- **Definition:** A **bind mount** is when you mount a directory or file from your host machine into a container. Docker does not manage it; it simply uses whatever path you specify on your host.
    
- **Location:** Any path on your host system, e.g., `/home/user/project`.

- **Use Cases:**
    - Develop locally (changes on host reflect immediately inside the container).
    - Access host files from inside the container.
        
- **How to use:**
    docker run -v /home/user/project:/app myimage
    Here, the host folder `/home/user/project` is mounted into the container at `/app`.
    
	**Advantages:**
	
	- Immediate access to host files.
	- Good for local development or debugging.
