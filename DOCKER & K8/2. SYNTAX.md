Here's a comprehensive guide to Docker syntax and how everything works:
## Docker CLI Syntax

The general pattern is: `docker [management-command] [subcommand] [options] [arguments]`

---

## Images wali shit : 

1. **Building an image** `docker build -t <name>:<tag> <context-path>` The `-t` flag tags the image. The context path (usually `.`) tells Docker which directory to send to the daemon. Docker reads a `Dockerfile` from that context by default. docker build -> creates images from the written dockerFiles. 

`docker build -f /path/to/Dockerfile .` lets you specify a custom Dockerfile location.

		1. **Listing images** `docker images` or `docker image ls` shows all local images with their repository, tag, ID, and size.

2. **Pulling/Pushing** `docker pull <image>:<tag>` downloads an image from a registry (Docker Hub by default). `docker push <image>:<tag>` uploads your image to a registry. You need to be logged in via `docker login`.

**Tagging** `docker tag <source-image> <target-image>:<tag>` creates an alias/tag for an image. similar to naming a commit message. 
 
**Removing** `docker rmi <image>` removes an image. Add `-f` to force-remove even if containers reference it.

---

## Containers

**Running a container** `docker run [options] <image> [command]`

Key flags:

- `-d` — detached (background) mode
- `-it` — interactive with a pseudo-TTY (for shell access)
- `--name <name>` — assign a name
- `-p <host-port>:<container-port>` — port mapping
- `-v <host-path>:<container-path>` — bind mount a volume
- `-e KEY=VALUE` — set environment variables
- `--env-file <file>` — load env vars from a file
- `--rm` — auto-remove the container when it exits
- `--network <network>` — attach to a specific network
- `--restart always|unless-stopped|on-failure` — restart policy
- `-u <user>` — run as a specific user
- `--cpus` / `--memory` — resource limits

Example: `docker run -d -p 8080:80 --name myapp nginx`

**Listing containers** `docker ps` shows running containers. `docker ps -a` shows all (including stopped).

**Starting/Stopping** `docker start <container>` starts a stopped container. `docker stop <container>` sends SIGTERM, then SIGKILL after a grace period. `docker kill <container>` sends SIGKILL immediately. `docker restart <container>` stops then starts.

**Executing commands in a running container** `docker exec -it <container> bash` opens an interactive shell inside a running container. `docker exec <container> <command>` runs a one-off command.

**Logs** `docker logs <container>` prints stdout/stderr. `-f` follows (tails) the log. `--tail 100` shows last 100 lines. `--since 10m` filters by time.

**Copying files** `docker cp <container>:<path> <host-path>` copies from container to host, and vice versa.

**Inspecting** `docker inspect <container|image>` returns detailed JSON metadata — networking, mounts, config, etc.

**Removing** `docker rm <container>` removes a stopped container. `-f` forces removal of a running one. `docker container prune` removes all stopped containers.

---

## Docker file Syntax

A Docker file is a script of instructions to build an image, executed top to bottom.

**FROM** <Image name>— base image (must be first instruction) `FROM ubuntu:22.04` `FROM scratch` starts from an empty image. Multi-stage: `FROM node:20 AS builder` 

**RUN** — executes a command during build `RUN apt-get update && apt-get install -y curl` Shell form: `RUN <command>` (runs via `/bin/sh -c`). Exec form: `RUN ["executable", "arg1"]` (no shell, no variable expansion). Each `RUN` creates a new layer, so chain commands with `&&` to keep layers small.

**COPY** — copies files from build context into image `COPY src/ /app/src/` `COPY --from=builder /app/dist ./dist` (multi-stage copy)

**ADD** — like COPY but also handles URLs and auto-extracts tar files Prefer COPY unless you specifically need ADD's extra features.

**WORKDIR** — sets the working directory for subsequent instructions `WORKDIR /app` Creates the directory if it doesn't exist.

**ENV** — sets environment variables `ENV NODE_ENV=production PORT=3000` These persist into the running container.

**ARG** — build-time variables (not persisted in final image) `ARG VERSION=1.0` Override at build time: `docker build --build-arg VERSION=2.0 .`

**EXPOSE** — documents which port the container listens on `EXPOSE 8080` This is informational only — it doesn't actually publish the port. You still need `-p` at runtime.

**CMD** — default command to run when container starts `CMD ["node", "server.js"]` (exec form — preferred) `CMD npm start` (shell form) Only one CMD is effective. Can be overridden at `docker run` time.

**ENTRYPOINT** — sets the main executable; CMD becomes its default arguments `ENTRYPOINT ["python3", "app.py"]` Harder to override than CMD. Use when the container has a single clear purpose. Combining: `ENTRYPOINT ["nginx"]` + `CMD ["-g", "daemon off;"]`

**VOLUME** — declares a mount point / anonymous volume `VOLUME /data`

**USER** — sets the user for subsequent instructions and the container process `USER node` — runs as non-root, which is a security best practice.

**LABEL** — adds metadata key-value pairs `LABEL maintainer="you@example.com" version="1.0"`

**HEALTHCHECK** — tells Docker how to check if the container is healthy `HEALTHCHECK --interval=30s --timeout=5s CMD curl -f http://localhost/ || exit 1`

**ONBUILD** — triggers an instruction when this image is used as a base `ONBUILD COPY . /app` — runs when someone does `FROM yourimage`.

**SHELL** — changes the default shell used by RUN/CMD/ENTRYPOINT shell form `SHELL ["/bin/bash", "-c"]`

**STOPSIGNAL** — sets the signal sent to stop the container `STOPSIGNAL SIGTERM`

## Volumes

Volumes are Docker-managed persistent storage, separate from the container filesystem.

`docker volume create <name>` creates a named volume. `docker volume ls` lists volumes. `docker volume inspect <name>` shows details. `docker volume rm <name>` removes a volume. `docker volume prune` removes all unused volumes.

Three types of mounts:

- **Named volume**: `-v mydata:/app/data` — managed by Docker, best for persistence.
- **Bind mount**: `-v /host/path:/container/path` — maps a host directory directly. Good for development.
- **tmpfs mount**: `--tmpfs /tmp` — stored in memory only, not persisted.

---

## Networks

`docker network create <name>` creates a user-defined bridge network. `docker network ls` lists networks. `docker network inspect <name>` shows details. `docker network connect <network> <container>` connects a running container to a network. `docker network disconnect <network> <container>` disconnects it. `docker network prune` removes unused networks. `docker network rm <name>` removes a network.

Network drivers:

- **bridge** — default for standalone containers; containers on the same bridge can communicate by container name.
- **host** — container shares the host's network stack (Linux only).
- **none** — no networking.
- **overlay** — multi-host networking for Docker Swarm.
- **macvlan** — assigns a MAC address to the container, making it appear as a physical device.

---

## Docker Compose

Compose uses a `docker-compose.yml` (or `compose.yaml`) file to define multi-container applications.

```yaml
version: "3.9"

services:
  web:
    build: .
    ports:
      - "8080:80"
    environment:
      - DEBUG=true
    depends_on:
      - db
    volumes:
      - ./app:/app
    networks:
      - backend

  db:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - pgdata:/var/lib/postgresql/data
    networks:
      - backend

volumes:
  pgdata:

networks:
  backend:
```

Key Compose commands:

- `docker compose up` — creates and starts all services. `-d` for detached. `--build` to force rebuild.
- `docker compose down` — stops and removes containers, networks. Add `-v` to also remove volumes.
- `docker compose build` — builds images only.
- `docker compose logs -f` — streams logs from all services.
- `docker compose ps` — lists running services.
- `docker compose exec <service> bash` — opens a shell in a service.
- `docker compose restart <service>` — restarts a specific service.
- `docker compose pull` — pulls latest images.
- `docker compose scale web=3` — scales a service to N replicas.

---

## System & Cleanup

`docker system df` shows disk usage by images, containers, volumes, and build cache. `docker system prune` removes all stopped containers, dangling images, unused networks, and build cache. `docker system prune -a` also removes unused (not just dangling) images. `docker system prune --volumes` includes unused volumes. `docker image prune` removes dangling images only. `docker builder prune` clears build cache.

---

## Registry & Login

`docker login` logs into Docker Hub (or a custom registry with `docker login myregistry.io`). `docker logout` logs out. `docker search <term>` searches Docker Hub for images.

---

## Multi-stage Builds

Multi-stage builds let you use multiple FROM instructions in one Dockerfile to keep the final image lean:

```dockerfile
# Stage 1: Build
FROM node:20 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Production image
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
CMD ["node", "dist/index.js"]
```

Only the final stage ends up in the image, so build tools and intermediate files are discarded.

## Kubernetes in the Docker Workflow

### Why Kubernetes Exists

Docker alone is great for running containers. But imagine you have 50 containers across 10 servers. Questions start appearing:

- What if a container crashes? Who restarts it?
- How do you roll out a new version without downtime?
- How do traffic get distributed across containers?
- How do you scale up when traffic spikes?

Manually doing this with `docker run` is a nightmare. **Kubernetes (K8s) is the answer** — it's a system that manages containers across multiple machines automatically.

---

### Mental Model

```
Docker    = runs a single container
Compose   = runs multiple containers on ONE machine
Kubernetes = runs and manages containers across MANY machines
```

Kubernetes treats your servers as a **pool of resources** and decides where to run things.

---

### Core Kubernetes Concepts

**Cluster** — The whole system. Made up of machines (called nodes).

**Node** — A single machine (physical or VM) in the cluster. There are two types: the **Control Plane** (the brain, manages everything) and **Worker Nodes** (where your containers actually run).

**Pod** — The smallest unit in K8s. A pod wraps one (or sometimes a few) containers. You never run containers directly — you run pods.

**Deployment** — Tells K8s "I want 3 copies of this pod running at all times." If one dies, K8s spins up a new one automatically.

**Service** — A stable network endpoint to reach your pods. Pods come and go, their IPs change — a Service gives them a fixed address.

**Ingress** — Manages external traffic coming into the cluster. Routes `yourapp.com/api` to one service, `yourapp.com/frontend` to another.

---

### How K8s Fits Into the Workflow

The workflow up to the registry stays exactly the same. K8s takes over after that:

```
push code
    │
    ▼
CI builds Docker image
    │
    ▼
pushes to Registry
    │
    ▼
Kubernetes pulls image          ← NEW from here
    │
    ▼
K8s schedules pods across nodes
    │
    ▼
Service routes traffic to pods
    │
    ▼
Ingress exposes app to internet 🚀
```

---

### What K8s Looks Like in Practice

Instead of `docker run`, you write **YAML manifest files** describing your desired state.

A **Deployment**:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3          # run 3 copies of this pod
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: my-app
          image: yourusername/my-app:v1.0   # from your registry
          ports:
            - containerPort: 5000
```

A **Service** to expose those pods internally:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
spec:
  selector:
    app: my-app        # targets pods with this label
  ports:
    - port: 80
      targetPort: 5000
```

Then you apply it:

```bash
kubectl apply -f deployment.yaml
kubectl apply -f service.yaml
```

K8s reads this and makes it happen. If a pod dies, K8s notices the actual state (2 pods) doesn't match desired state (3 pods) and creates a new one. This is called the **reconciliation loop** and it's the core idea behind K8s.

---

### Key Things K8s Gives You

**Self-healing** — Crashed container? K8s restarts it automatically.

**Scaling** — Need more capacity? One command:

```bash
kubectl scale deployment my-app --replicas=10
```

Or set up **auto-scaling** based on CPU usage — K8s scales up and down on its own.

**Rolling updates** — Deploy a new image version with zero downtime. K8s replaces pods one by one, keeping the app alive the whole time:

```bash
kubectl set image deployment/my-app my-app=yourusername/my-app:v2.0
```

**Rollback** — New version broken? Roll back instantly:

```bash
kubectl rollout undo deployment/my-app
```

---

### The Honest Reality

Kubernetes is powerful but has a steep learning curve and real operational overhead. You don't need it until you do.

|Situation|What to use|
|---|---|
|Side project / small app|Docker Compose on a VPS|
|Growing startup, a few servers|Maybe Docker Swarm, or managed K8s|
|Large scale, many teams, microservices|Kubernetes|

Most teams use **managed Kubernetes** — meaning the cloud provider runs the control plane for you. You just manage your workloads. Examples: **GKE** (Google), **EKS** (AWS), **AKS** (Azure).
