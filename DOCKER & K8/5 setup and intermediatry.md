### 1. VOLUME AND PERSISTENT STORAGES 

https://www.geeksforgeeks.org/devops/what-is-docker-volume/
[Docker Volumes] are persistent data stores for containers, created and managed by Docker, They provide a reliable and efficient way to ensure data persistence when working with containers. Docker volumes are file systems that are mounted on [Docker containers](https://www.geeksforgeeks.org/blogs/containerization-using-docker/) to preserve the data generated by the container.
By design the docker storage are ephemeral ie temporary and are destroyed when the docker containers are stop. 

In a typical workflow, if you want to apply changes to a container, you:

1. Update the Docker image,
2. Remove the old container, and
3. Launch a new one from the updated image.

While this approach works well for **stateless applications**, it becomes problematic for \n containers that need to **retain state** such as **databases** (e.g, MySQL, PostgreSQL) or services that generate or store user data.

Docker **volumes** solve this problem by allowing you to ****store data outside the container'sfilesystem****, in a persistent and reusable location managed by Docker. Even if the container is stopped, removed, or rebuilt, the data in the volume remains intact.```

- Data durability across container restarts or replacements.
- Separation of application logic (in containers) from persistent data (in volumes).
- A scalable and production-ready way to manage storage for stateful applications.
docker network : creating isolated network of docker containers. so that multiple docker containers can be connected

**Named and Anoynomus volumes**

Docker supports two types of volumes: ****named**** and *anonymous*. While both serve the same purpose—persisting data outside of a container’s lifecycle—they behave differently when it comes to management and reuse.
  
****Named Volumes****
- Created with a specific name you assign.
- Easy to reference and reuse across multiple containers.
- Ideal for ****persistent data**** that needs to be accessed or managed long-term (e.g., db-data, app-config). 
	- creating : *docker volume create my_volume*
	- referencing : *docker run -d --name my_container -v my_volume:/app/data nginx*

 ****Anonymous Volumes****
- Automatically assigned a ****random name**** (e.g, 13f6a8c5b6b7e0e22349c1...) by Docker.
- Useful for ****temporary storage**** when you don’t need to manage the volume manually.
- Persist even after the container is removed, unless the container was run with the --rm flag, in which case the volume is deleted along with the container.
- each container has its own anoynmus volume. To share an anonymous volume, you must explicitly use its ****generated volume ID**** when running other containers.
	- command and referecning in the volume: *docker run -d --name my_container -v app/data nginx*

## DOCKER Networking -> Docker Network

```
docker network ls # lists all the networks
```

ways for containers to talk to each other. Docker creates **virtual networks** to connect containers. Then we can put the containers in the same network for the containers to communicate with each other easily. 

create a network : *docker network create <name>*
Ex : 
1 CREATE A MONGO DATABASE :
```docker network create mongo_net``

2. Creating a docker container to run the express and mongo connection : 
```docker run -d -p8081:8081 --name mongo_exp --network mongo_net -e MONGO_INITDB_ROOT_USERNAME=admin -e MONGO_INITDB_ROOT_PASSWORD=1234 -e ME_CONFIG_MONGODB_SERVER=mongo mongo-express```

3. Creating a docker container mongo database : 
```docker run -d -p27017:27017 --name mongo --network mongo_net -e MONGO_INITDB_ROOT_USERNAME=admin -e MONGO_INITDB_ROOT_PASSWORD=1234 mongo```

### *Docker networking modes* : 
![[Pasted image 20260225115928.png]]

Docker provides several **networking modes** that control how containers communicate with each other, the host, and external networks

BRIDGE : [dafault] : Each container gets its own private IP address on a virtual network created by Docker. Containers on the same bridge network can communicate via IP or container name. This is the default mode if you don’t specify a network. 

HOST : The container shares the host’s network stack. No isolation: container ports are the same as host ports

OVERLAY : Allows containers running on different Docker hosts (nodes in a Swarm) to communicate securely. Uses VXLAN encapsulation.

MACVLAN : containers appers as a seperate physical devices for some containers thr needs the sperate MAC and IP addereces. Containers appear as physical devices on the network, with their own MAC and IP addresses. Useful for legacy apps that expect to be on a physical network.

NONE : The container has **no networking**. Useful for highly restricted containers or testing purposes.

## Docker Compose : 

Docker Compose is a tool for defining and running multi-container applications. It is the key to unlocking a streamlined and efficient development and deployment experience. docker compose simplifies the control of entire applications, Then, with a single command, you create and start all the services from your configuration file. 

*Key benefits of Docker Compose*

Using Docker Compose offers several benefits that streamline the development, deployment, and management of containerized applications.
1. Simplified control. (single fine has the docker setup for the entire application)
2. Efficient collaboration. (no need to edit and manual setup.)
3. Rapid application development. 
4. Portability across environments. (promotes)
