> Capture and store an object's **internal state** at a point in time, so it can be **restored later** — without violating encapsulation. The object itself controls what gets saved and restored. This point is important to not violate the encapsulation principal of the pilars.

- **Originator:** The object whose state needs to be saved/restored. 
- **Memento:** A snapshot object that stores the state. 
- **Caretaker:** Manages the history of mementos (save, undo, redo) but never inspects the memento's contents. 

**Idea:** Capture and store an object's **internal state** at a point in time, so it can be **restored later** — without violating encapsulation. The object itself controls what gets saved and restored.
### Structure
- **Originator:** The object whose state needs to be saved/restored
- **Memento:** A snapshot object that stores the state (treated as a black box)
- **Caretaker:** Manages the history of mementos (save, undo, redo) but never inspects the memento's contents

### Python Example — Game Character with Save/Load System

```python
from __future__ import annotations
from dataclasses import dataclass
from copy import deepcopy

# ── Memento ───────────────────────────────────────────────
@dataclass
class GameMemento:
    """Black box — Caretaker stores this but never reads it"""
    _level:  int
    _health: int
    _weapon: str
    _position: tuple

    def get_state(self):
        return {
            "level":    self._level,
            "health":   self._health,
            "weapon":   self._weapon,
            "position": self._position
        }

class GameCharacter:
    def __init__(self, name):
        self.name     = name
        self._level   = 1
        self._health  = 100
        self._weapon  = "Sword"
        self._position = (0, 0)

    def play(self, level, health, weapon, position):
        """Simulate game progression"""
        self._level    = level
        self._health   = health
        self._weapon   = weapon
        self._position = position

    def save(self) -> GameMemento:
        """Create a snapshot of current state"""
        print(f"  [SAVE] {self}")
        return GameMemento(
            self._level,
            self._health,
            self._weapon,
            deepcopy(self._position)
        )

    def restore(self, memento: GameMemento):
        """Restore state from a snapshot"""
        state          = memento.get_state()
        self._level    = state["level"]
        self._health   = state["health"]
        self._weapon   = state["weapon"]
        self._position = state["position"]
        print(f"  [LOAD] {self}")

    def __str__(self):
        return (f"{self.name} | Level {self._level} | "
                f"HP:{self._health} | Weapon:{self._weapon} | "
                f"Pos:{self._position}")

class SaveManager:
    def __init__(self):
        self._slots: dict[str, GameMemento] = {}   # named save slots
        self._history: list[GameMemento]    = []   # undo stack

    def save_to_slot(self, name: str, memento: GameMemento):
        self._slots[name] = memento
        self._history.append(memento)
        print(f"  Saved to slot: '{name}'")

    def load_slot(self, name: str) -> GameMemento:
        if name not in self._slots:
            raise KeyError(f"No save found in slot '{name}'")
        return self._slots[name]

    def undo(self) -> GameMemento | None:
        if len(self._history) < 2:
            print("  Nothing to undo")
            return None
        self._history.pop()              # remove current
        return self._history[-1]         # return previous

    def list_slots(self):
        print(f"  Save Slots: {list(self._slots.keys())}")


# ── Client ────────────────────────────────────────────────
hero    = GameCharacter("Arthas")
manager = SaveManager()

print("=== Starting Game ===")
print(f"  Initial: {hero}")

print("\n=== Checkpoint 1: Entering Forest ===")
hero.play(level=2, health=95, weapon="Sword", position=(10, 5))
manager.save_to_slot("forest_entry", hero.save())

print("\n=== Checkpoint 2: Boss Fight ===")
hero.play(level=5, health=80, weapon="Excalibur", position=(50, 30))
manager.save_to_slot("before_boss", hero.save())

print("\n=== Boss kills the hero (HP = 0) ===")
hero.play(level=5, health=0, weapon="Excalibur", position=(55, 35))
print(f"  Current: {hero}")

print("\n=== Load Last Save ===")
hero.restore(manager.load_slot("before_boss"))

print("\n=== Undo to Previous Save ===")
previous = manager.undo()
if previous:
    hero.restore(previous)

print("\n=== All Save Slots ===")
manager.list_slots()
```

### How Data Flows

```
Client
  │
  ├──► Originator.save() ──────────► creates Memento (snapshot)
  │                                        │
  │                                        ▼
  └──► Caretaker stores Memento ◄──────────┘
            │
            └──► Originator.restore(memento) ──► state rolls back
```

### Memento vs Command for Undo

| Feature       | Memento               | Command                 |
| ------------- | --------------------- | ----------------------- |
| Undo strategy | Restore full snapshot | Reverse each operation  |
| Best for      | Complex state objects | Simple discrete actions |
| Memory usage  | Higher (full copies)  | Lower (just operations) |
| Encapsulation | Strong                | Moderate                |
|               |                       |                         |

---

### When to Use

- You need **undo/redo** or **rollback** functionality
- You want to take **snapshots** without exposing internal state
- You need **multiple named save points** (like game checkpoints)
- Implementing **transactional operations** with rollback on failure

### Real-World Examples

- **Game save systems** — checkpoint and restore game state
- **Text editors** — undo history (Ctrl+Z)
- **Database transactions** — savepoints and rollback
- **Version control (Git)** — commits are essentially mementos
- **Form wizards** — go back to a previous step with data intact
- **Browser history** — back/forward navigation

### Pros & Cons

|Pros|Cons|
|---|---|
|Preserves encapsulation cleanly|Memory heavy with large states|
|Simple undo/redo implementation|Caretaker must manage lifecycle|
|Originator controls what is saved|Deep copies can be expensive|
|Multiple save slots easily supported|No partial state saves|

## Command vs Memento -> really important :

Both patterns support **undo/redo** but they think about it completely differently. This is the key source of confusion.
### Core Philosophy Difference

||Command|Memento|
|---|---|---|
|**Mindset**|_"Record what was done, reverse it"_|_"Photograph the state, restore it"_|
|**Undo approach**|Execute an inverse operation|Rewind to a saved snapshot|
|**Knows about logic?**|Yes — knows HOW to undo|No — just stores raw state|


### 1. Command Pattern
> _"I know **what you did** and I know **how to reverse it**"_

- Tracks every action as an object
- Each command **carries its own undo logic** (the math, the inverse operation)
- Can walk back through history **step by step**
- The command has to **peek inside** the object to reverse things — so encapsulation takes a slight hit
### 2. Memento Pattern

> _"I don't know what you did, I just **took a photo** before you did it"_

- Takes a **blind snapshot** of the full object state
- Has **zero knowledge** of what changed or why
- Restores by simply **putting the photo back**
- The object saves its **own** state — so nobody outside ever sees the internals ✅ encapsulation fully preserved
### The One-Line Distinction
||Command|Memento|
|---|---|---|
|**How it undoes**|_"Let me reverse what I did"_|_"Let me forget what happened"_|
