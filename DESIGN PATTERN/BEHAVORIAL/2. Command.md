**Command** is a behavioral design pattern that turns a request into a stand-alone object that contains all information about the request. This transformation lets you pass requests as a method arguments, delay or queue a request’s execution, and support undoable operations. 
Encapsulate a **request as an object**. This lets you parameterize actions, queue them, log them, or support **undo/redo** operations. 

General order of shit : 
- **Command (Abstract):** Interface with `execute()` and `undo()` methods
- **Concrete Command:** Implements the action, holds reference to the receiver
- **Receiver:** The object that actually performs the work
- **Invoker:** Triggers the command. 
- **Client:** Creates and configures commands


WHY? 
1. *Decoupling* : the caller does not need to know the internal workings of the execution of command.  Also promotes the seperation of concerns. 
	 *example* - Button → triggers command
			Command → executes device logic.
			Button doesn't know anything about the device.
2. *Easy Undo/Redo* : Undo redo functionality in the bag. Transaction wali shit and rollbacks are now available. 
3. *Queueing* : Commands can be queud and executed later and schedulled. Task queue, job queues and background workers. 
4. *Commandic macros* : Combining commands. ( start -> set -> start )



EXAMPLE : 
```python 
from abc import ABC, abstractmethod
# ── Abstract Command ──────────────────────────────────────
class Command(ABC):
    @abstractmethod
    def execute(self):
        pass
    @abstractmethod
    def undo(self):
        pass 
        
class TextEditor:
    def __init__(self):
        self.text = ""
    def write(self, words):
        self.text += words
        print(f"Editor: '{self.text}'")
    def erase(self, num_chars):
        self.text = self.text[:-num_chars]
        print(f"Editor: '{self.text}'")

class WriteCommand(Command):
    def __init__(self, editor, words):
        self.editor = editor
        self.words = words
    def execute(self):
        self.editor.write(self.words)
    def undo(self):
        self.editor.erase(len(self.words))

class EraseCommand(Command):
    def __init__(self, editor, num_chars):
        self.editor = editor
        self.num_chars = num_chars
        self.erased = ""       
    def execute(self):
        self.erased = self.editor.text[-self.num_chars:]
        self.editor.erase(self.num_chars)
    def undo(self):
        self.editor.write(self.erased)
        
class CommandManager:
    def __init__(self):
        self.history = []
    def execute(self, command):
        command.execute()
        self.history.append(command)
    def undo(self):
        if not self.history:
            print("Nothing to undo")
            return
        command = self.history.pop()
        command.undo()

editor  = TextEditor()
manager = CommandManager()

print("=== Executing Commands ===")
manager.execute(WriteCommand(editor, "Hello, "))
manager.execute(WriteCommand(editor, "World!"))
manager.execute(EraseCommand(editor, 6))       # erase "World!"

print("\n=== Undoing Commands ===")
manager.undo()   # restore "World!"
manager.undo()   # remove "World!"
manager.undo()   # remove "Hello, "
manager.undo()   # nothing left
```

**Output:**
```
=== Executing Commands ===
Editor: 'Hello, '
Editor: 'Hello, World!'
Editor: 'Hello, '

=== Undoing Commands ===
Editor: 'Hello, World!'
Editor: 'Hello, '
Editor: ''
Nothing to undo
```
### How Data Flows
```
Client
  │
  │ creates ──► ConcreteCommand (holds Receiver + params)
  │
  └──► Invoker.execute(command)
            │
            ├── command.execute() ──► Receiver does the work
            └── stores in history   (for undo later)
```

When to use : 
- You need **undo/redo** functionality
- You want to **queue or schedule** operations
- You want to **log or audit** all actions taken
- You need to support **transactional behavior** (rollback on failure)
--- 

REAL WORLD EXAMPLE : 
- **Text editors** — Ctrl+Z / Ctrl+Y (undo/redo).
- **Database transactions** — commit and rollback.
- **Job queues** — Celery tasks are essentially commands.
- **GUI buttons** — each button encapsulates an action object and frontend. 
- **Macro recording** — record a sequence of commands and replay.
