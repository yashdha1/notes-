
**Singleton** is a creational design pattern that lets you ensure that a class has only one instance, while providing a global access point to this instance.  

A **Singleton** ensures a class has **only one instance** and provides a **global access point** to it.

```
Problem:  Some resources must exist exactly once
          (config, logger, DB connection pool, cache)

Solution: The class itself controls instantiation,
          returning the same object every time
```

---

## Python Implementations

### 1. Classic — `__new__` override

```python
class Singleton:
    _instance = None

    def __new__(cls, *args, **kwargs):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance


a = Singleton()
b = Singleton()
print(a is b)  # True — exact same object
```

### 2. Thread-Safe Singleton

Without locking, two threads can **simultaneously** pass the `None` check and create two instances:

```python
import threading

class ThreadSafeSingleton:
    _instance = None
    _lock = threading.Lock()

    def __new__(cls):
        if cls._instance is None:          # first check (no lock — fast path)
            with cls._lock:
                if cls._instance is None:  # second check (with lock — safe)
                    cls._instance = super().__new__(cls)
        return cls._instance
```

This is the **double-checked locking** pattern — avoids lock overhead on every access after first creation.
### 3. Metaclass Singleton

Cleaner separation — the singleton logic lives in the metaclass, not the class itself:

```python
class SingletonMeta(type):
    _instances = {}
    _lock = threading.Lock()

    def __call__(cls, *args, **kwargs):
        if cls not in cls._instances:
            with cls._lock:
                if cls not in cls._instances:
                    instance = super().__call__(*args, **kwargs)
                    cls._instances[cls] = instance
        return cls._instances[cls]


class AppConfig(metaclass=SingletonMeta):
    def __init__(self):
        self.debug = False
        self.db_url = "postgresql://localhost/app"
        self.secret_key = "abc123"

class Logger(metaclass=SingletonMeta):
    def __init__(self):
        self.level = "INFO"


# Each class gets its own single instance
config = AppConfig()
logger = Logger()
```

### 4. Module-Level Singleton (Most Pythonic)

Python **modules are singletons by default** — imported once and cached in `sys.modules`:

```python
# config.py
class _Config:
    def __init__(self):
        self.debug = False
        self.db_url = "postgresql://localhost/app"
        self.max_connections = 10

    def load_from_env(self):
        import os
        self.debug = os.getenv("DEBUG", "false") == "true"
        self.db_url = os.getenv("DATABASE_URL", self.db_url)

# Module-level instance — created once, shared everywhere
config = _Config()
```

```python
# anywhere in your app
from config import config   # same object every import

config.debug = True
```

This is the **recommended Python approach** for most use cases — no boilerplate, no locks needed.

### 5. Borg Pattern (Shared State, Not Identity)

Instead of sharing the _instance_, share the _state_. All instances behave as one:

```python
class Borg:
    _shared_state = {}  # class-level shared dict

    def __init__(self):
        self.__dict__ = self._shared_state  # all instances point here


class AppSettings(Borg):
    def __init__(self):
        super().__init__()
        if not hasattr(self, "initialized"):
            self.theme = "dark"
            self.language = "en"
            self.initialized = True


a = AppSettings()
b = AppSettings()

a.theme = "light"
print(b.theme)   # "light" — shared state
print(a is b)    # False — different objects, same state
```

Borg is more **subclass-friendly** than true Singleton — state is shared but identity is not enforced.

## Real-World Usage Examples

### Logger

```python
import logging
import threading

class AppLogger(metaclass=SingletonMeta):
    def __init__(self):
        self._logger = logging.getLogger("app")
        handler = logging.StreamHandler()
        handler.setFormatter(logging.Formatter(
            "%(asctime)s [%(levelname)s] %(message)s"
        ))
        self._logger.addHandler(handler)
        self._logger.setLevel(logging.DEBUG)

    def info(self, msg):  self._logger.info(msg)
    def error(self, msg): self._logger.error(msg)
    def debug(self, msg): self._logger.debug(msg)


# Same logger instance, everywhere in your app
log = AppLogger()
log.info("Server started")
```

### DB Connection Pool

```python
import threading
from contextlib import contextmanager

class DatabasePool(metaclass=SingletonMeta):
    def __init__(self, url: str, max_connections: int = 10):
        self.url = url
        self.max_connections = max_connections
        self._pool = []
        self._lock = threading.Lock()
        self._initialize_pool()

    def _initialize_pool(self):
        # Create connections once
        for _ in range(self.max_connections):
            self._pool.append(self._create_connection())

    def _create_connection(self):
        return {"url": self.url, "active": False}  # simplified

    @contextmanager
    def get_connection(self):
        conn = self._acquire()
        try:
            yield conn
        finally:
            self._release(conn)

    def _acquire(self):
        with self._lock:
            for conn in self._pool:
                if not conn["active"]:
                    conn["active"] = True
                    return conn
        raise RuntimeError("No connections available")

    def _release(self, conn):
        with self._lock:
            conn["active"] = False


# One pool, shared across all threads
pool = DatabasePool("postgresql://localhost/app", max_connections=5)

with pool.get_connection() as conn:
    print(f"Using: {conn}")
```

## The Controversy — When Singleton Becomes an Anti-Pattern

Singleton is **one of the most debated patterns** in SWE. Here's why:

### Problems in Practice

```
1. Global state         →  any code can mutate shared state unpredictably
2. Hidden dependencies  →  callers don't declare they need it (invisible coupling)
3. Testability          →  hard to mock/replace; state bleeds between tests
4. Concurrency          →  shared mutable state = race conditions if not careful
5. SRP violation        →  class manages its own lifecycle AND does its job
6. Tight coupling       →  code that uses Singleton is hard to reuse elsewhere
```

### The Testing Problem in Detail

```python
# BAD — Singleton bleeds state between tests
class Cache(metaclass=SingletonMeta):
    def __init__(self):
        self.store = {}

def test_cache_stores_value():
    c = Cache()
    c.store["key"] = "value"
    assert c.store["key"] == "value"

def test_cache_is_empty():
    c = Cache()
    assert c.store == {}   # ❌ FAILS — previous test left "key" behind!
```

```python
# GOOD — Dependency injection, easy to test
class Cache:
    def __init__(self):
        self.store = {}

def test_cache_is_empty():
    c = Cache()              # fresh instance, no shared state
    assert c.store == {}     # always passes
```

## Singleton vs Dependency Injection

The modern SWE preference is **DI over Singleton** for most cases:

```python
# Singleton approach — hidden, global, hard to test
class UserService:
    def get_user(self, id):
        db = DatabasePool()   # hidden dependency — tight coupling
        ...

# DI approach — explicit, flexible, testable
class UserService:
    def __init__(self, db: DatabasePool):   # dependency declared upfront
        self.db = db

    def get_user(self, id):
        ...

# In tests — swap with a mock effortlessly
mock_db = MockDatabase()
service = UserService(db=mock_db)
```

## Decision Guide

```
Ask yourself:
│
├── Is there a genuine physical constraint to one instance?
│   (hardware device, single file, OS resource)
│   └── YES → Singleton may be appropriate
│
├── Do I just want convenient global access?
│   └── YES → Use dependency injection instead
│
├── Is this for configuration/settings?
│   └── YES → Python module-level singleton is enough
│
├── Will this be unit tested?
│   └── YES → Avoid Singleton, prefer DI
│
└── Is this shared across threads?
    └── YES → Use double-checked locking or module singleton
```

## Pattern Comparison at a Glance

|Approach|Thread-Safe|Pythonic|Testable|Use When|
|---|---|---|---|---|
|`__new__`|❌ No|Moderate|Hard|Simple scripts|
|Double-checked lock|✅ Yes|Moderate|Hard|Multi-threaded apps|
|Metaclass|✅ Yes|Clean|Hard|Frameworks, libraries|
|Module-level|✅ Yes|✅ Most|Moderate|Most Python projects|
|Borg|✅ Yes|Unique|Moderate|Shared state, flexible identity|
|DI (not singleton)|✅ Yes|✅ Best|✅ Easy|Production code, tested apps|

---

## The Bottom Line

> Singleton solves a **real problem** — shared, single-instance resources — but the implementation often creates **more problems than it solves** through global state and hidden coupling.

In modern Python: **prefer module-level singletons or dependency injection**. Reserve the full Singleton pattern for cases where the uniqueness constraint is a genuine requirement, not just a convenience.
