
The **Decorator** pattern **dynamically adds behavior to an object** without modifying its class. It wraps objects in decorator objects that add new functionality.
#### Concept
> "Attach additional responsibilities to an object at runtime — a flexible alternative to subclassing."

#### Real-World Analogy
A **coffee order** — you start with plain coffee, then wrap it with milk, sugar, whipped cream... each addition wraps the previous one.

### Example

```python
from abc import ABC, abstractmethod

# --- COMPONENT - base interface ---
class Coffee(ABC):
    @abstractmethod
    def cost(self) -> float:
        pass
    @abstractmethod
    def description(self) -> str:
        pass
        
# --- CONCRETE COMPONENT - the base object ---
class SimpleCoffee(Coffee):
    def cost(self) -> float:
        return 1.00
    def description(self) -> str:
        return "Simple Coffee"

# --- BASE DECORATOR - wraps a Coffee object ---
class CoffeeDecorator(Coffee):
    def __init__(self, coffee: Coffee):
        self._coffee = coffee          # holds reference to wrapped object
    def cost(self) -> float:
        return self._coffee.cost()     # delegates to wrapped object
    def description(self) -> str:
        return self._coffee.description()


# --- CONCRETE DECORATORS - each adds something new ---
class Milk(CoffeeDecorator):
    def cost(self) -> float:
        return self._coffee.cost() + 0.25

    def description(self) -> str:
        return self._coffee.description() + " + Milk"


class Sugar(CoffeeDecorator):
    def cost(self) -> float:
        return self._coffee.cost() + 0.10

    def description(self) -> str:
        return self._coffee.description() + " + Sugar"


class WhippedCream(CoffeeDecorator):
    def cost(self) -> float:
        return self._coffee.cost() + 0.50

    def description(self) -> str:
        return self._coffee.description() + " + Whipped Cream"


# --- Usage - stack decorators at runtime! ---
coffee = SimpleCoffee()
print(f"{coffee.description():<45} ${coffee.cost():.2f}")

coffee = Milk(coffee)
print(f"{coffee.description():<45} ${coffee.cost():.2f}")

coffee = Sugar(coffee)
print(f"{coffee.description():<45} ${coffee.cost():.2f}")

coffee = WhippedCream(coffee)
print(f"{coffee.description():<45} ${coffee.cost():.2f}")

# You can also stack the same decorator multiple times!
coffee = Sugar(coffee)
print(f"{coffee.description():<45} ${coffee.cost():.2f}")
```

### Output

```
Simple Coffee                                 $1.00
Simple Coffee + Milk                          $1.25
Simple Coffee + Milk + Sugar                  $1.35
Simple Coffee + Milk + Sugar + Whipped Cream  $1.85
Simple Coffee + Milk + Sugar + Whipped Cream + Sugar $1.95
```

### Wrapping Visualization

```
Sugar
  └── WhippedCream
        └── Milk
              └── SimpleCoffee  ← core object
```

Each layer wraps the previous, delegating and adding behavior.
### Python's Built-in Decorators
Python has native decorator syntax — a different but related concept:

```python
# Python's @decorator syntax is inspired by this pattern
def log(func):
    def wrapper(*args, **kwargs):
        print(f"Calling {func.__name__}...")
        result = func(*args, **kwargs)
        print(f"Done.")
        return result
    return wrapper

@log
def greet(name):
    print(f"Hello, {name}!")

greet("Alice")
# Calling greet...
# Hello, Alice!
# Done.
```

### Key Roles

|Role|In Example|
|---|---|
|**Component**|`Coffee` (base interface)|
|**Concrete Component**|`SimpleCoffee` (base object)|
|**Base Decorator**|`CoffeeDecorator` (wraps component)|
|**Concrete Decorators**|`Milk`, `Sugar`, `WhippedCream`|

### Decorator vs Inheritance
||Decorator|Inheritance|
|---|---|---|
|**When**|Runtime|Compile time|
|**Flexibility**|Stack freely|Fixed hierarchy|
|**Classes needed**|Few|Explodes quickly|

### When to Use
- You want to **add behavior without modifying** the original class
- You need **flexible combinations** of features at runtime
- Subclassing would cause a **class explosion**
- Great for **logging, caching, authentication, validation**
