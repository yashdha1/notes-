
Replacing *Inheritance* with *Composition*. 

Bridge** is a structural design pattern that lets you split a large class or a set of closely related classes into two separate hierarchies—abstraction and implementation—which can be developed independently of each other. It replaces inheritance with **COMPOSITION**. 

> has-a relationship.

Rather than having subclasses having to implement subtypes. indivisually. Bridge Pattern
The **Bridge** pattern **decouples an abstraction from its implementation** so that the two can vary independently. It replaces inheritance with composition.

> "Separate the  *[is-a]* _what_ (abstraction) from the *[has-a]* _how_ (implementation)."

### The Problem Without Bridge

```python
# Without Bridge - explosion of subclasses!
class RedCircle: ...
class BlueCircle: ...
class RedSquare: ...
class BlueSquare: ...
# Adding a new shape OR color = many new classes 
```

### Example

```python
# --- IMPLEMENTATION side (the "how") ---
class Color:
    def fill(self) -> str:
        raise NotImplementedError

class Red(Color):
    def fill(self) -> str:
        return "Red"

class Blue(Color):
    def fill(self) -> str:
        return "Blue"


# --- ABSTRACTION side (the "what") ---
class Shape:
    def __init__(self, color: Color):  # Bridge: holds a reference to implementation
        self._color = color

    def draw(self) -> str:
        raise NotImplementedError

class Circle(Shape):
    def draw(self) -> str:
        return f"Drawing a {self._color.fill()} Circle"

class Square(Shape):
    def draw(self) -> str:
        return f"Drawing a {self._color.fill()} Square"


# Usage - mix and match freely!
red_circle  = Circle(Red())
blue_square = Square(Blue())
blue_circle = Circle(Blue())

print(red_circle.draw())   # Drawing a Red Circle
print(blue_square.draw())  # Drawing a Blue Square
print(blue_circle.draw())  # Drawing a Blue Circle
```

### Key Roles

| Role                     | In Example         |     |
| ------------------------ | ------------------ | --- |
| **Abstraction**          | `Shape`            |     |
| **Refined Abstraction**  | `Circle`, `Square` |     |
| **Implementor**          | `Color`            |     |
| **Concrete Implementor** | `Red`, `Blue`      |     |

### Adapter vs Bridge
||Adapter|Bridge|
|---|---|---|
|**Purpose**|Fix incompatible interfaces _after the fact_|Design for independence _upfront_|
|**Focus**|Compatibility|Scalability|
|**Timing**|Retrofit|Planned|

### When to Use
- You want to avoid a **cartesian product** explosion of subclasses
- Both **abstractions and implementations** should be extensible via subclassing
- You want to **switch implementations at runtime**
