
## **Persisting data to a database**

- **Problem:** With RDB snapshots or AOF, Redis might still lose recent messages (between snapshots or fsyncs).
- **Solution:** Write messages to a **durable database** at the same time you write to Redis Streams.
    - Example flow:
        Producer -> Redis Stream (for fast pub/sub) + DB (for durability)
        
    - This ensures:
        - Fast delivery to consumers via Redis Streams
        - Safe storage in DB (like PostgreSQL, MongoDB, or Cassandra)
    - You can then **replay messages from DB** if Redis crashes.
        
- **Trade-offs:**
    
    - Slightly higher latency for writes (writing twice)
    - More complex architecture
    - But essentially **guarantees no data loss**
This is very common in production for **critical financial/order/event systems**.

## **Multiple Redis servers (Replication / Cluster)**

- **Replication:**
    - Use **1 master + 1 or more replicas**
    - Master handles writes; replicas copy data
    - If master crashes, a replica can take over (with **Sentinel** for failover)
        
- **Persistence options:**
    - AOF + replication = almost no data loss
    - RDB + replication = some data loss possible
        
- **Cluster mode:**
    - Shards data across multiple nodes
    - Each shard can have replicas
    - Provides **both high availability and scalability**
        
- **Trade-offs:**
    - More infrastructure (more Redis instances)
    - Some complexity in failover management
    - Needs monitoring to ensure replicas are in sync
Many companies rely **only on Redis replication + AOF** for near-real-time systems where a few seconds of potential loss is acceptable.
